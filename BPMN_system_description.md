# Опис системи "Читальня" для генерації BPMN діаграми

## Загальний контекст

**"Читальня"** — веб-платформа для читання та прослуховування книг.  
Мови: TypeScript (React фронтенд) + Python (FastAPI бекенд) + PostgreSQL.  
Розгортання: Render (бекенд), Vercel (фронтенд).

---

## Учасники (Pools / Lanes)

| Учасник | Роль |
|---------|------|
| **Гість** | Не авторизований відвідувач |
| **Користувач** | Зареєстрований читач (роль: `user`) |
| **Автор** | Користувач з роллю `author`, публікує книги |
| **Адміністратор** | Модерує контент, керує платформою |
| **Зовнішні системи** | Stripe, BookNet.ua, Open Library, Gutendex, LibriVox, Edge-TTS |

## Процес 1: Реєстрація та авторизація

### Старт: Гість відкриває сайт

**Потік реєстрації:**
1. Гість заповнює форму (email, username, password)
2. `POST /api/auth/register`
3. **[Шлюз]** Email вже зайнятий?
   - Так → повернути помилку 400
   - Ні → хешувати пароль (bcrypt)
4. Створити запис `User` (роль = `user`)
5. Автоматично створити `UserSubscription` (план = `free`)
6. Повернути пару токенів (Access JWT 30 хв + Refresh JWT 7 днів)
7. Клієнт зберігає токени у localStorage / Zustand store

**Потік входу:**
1. Гість вводить email + password → `POST /api/auth/login`
2. **[Шлюз]** Акаунт активний та пароль вірний?
   - Ні → 401 Unauthorized
   - Так → повернути пару токенів

**Оновлення токена:**
1. Access токен прострочений → клієнт надсилає `POST /api/auth/refresh`
2. **[Шлюз]** Refresh токен валідний?
   - Ні → перенаправити на логін
   - Так → нова пара токенів

**Відновлення паролю:**
1. Користувач надсилає email → `POST /api/auth/forgot-password`
2. Система генерує токен (SHA-256), зберігає хеш із TTL 30 хв
3. Надсилає email з посиланням
4. Користувач переходить за посиланням → `POST /api/auth/reset-password`
5. **[Шлюз]** Токен валідний та не використаний?
   - Ні → помилка
   - Так → оновити пароль, токен позначити як використаний

## Процес 2: Пошук та перегляд каталогу

### Старт: Користувач відкриває каталог

1. Клієнт надсилає `GET /api/books?q=...&sort=...&offset=...`
2. **[Шлюз]** Пошуковий запит наявний та offset=0?
   - Так → `live_sync_books()`: паралельний запит до Open Library + Gutendex
     - Знайдені книги зберігаються/оновлюються у БД
   - Ні → пропустити зовнішню синхронізацію
3. Фільтрувати (виключити мову `ru`, виключити статус `draft`/`banned`)
4. Сортувати (new / popular / rating)
5. Повернути список книг з пагінацією
6. Клієнт відображає картки книг

---

## Процес 3: Доступ до книги та читання/прослуховування

### Старт: Користувач натискає на книгу

**Перевірка доступу:**
1. Клієнт → `GET /api/books/{id}/access`
2. Бекенд перевіряє в порядку пріоритету:
   - Адмін → `admin`
   - Власник книги → `owner`
   - Книга безкоштовна → `free`
   - Активна підписка на платформу (`platform_premium`) → `platform_premium`
   - Активна підписка на автора → `author_sub`
3. **[Шлюз]** Доступ отримано?
   - Ні → показати PaymentModal → перейти до Процесу 5
   - Так → дозволити читання/прослуховування

**Читання тексту (`GET /api/books/{id}/stream/text`):**
1. Перевірити доступ
2. **[Шлюз]** Джерело тексту:
   - `text_content` у БД → повернути напряму
   - Локальний файл (`text_path`) → читати з диска
   - `text_url` → проксі-запит до зовнішнього URL
   - Нічого → Gutendex API (пошук за назвою) → завантажити та закешувати
3. Клієнт рендерить текст у Reader

**Прослуховування аудіо (`GET /api/books/{id}/stream/audio`):**
1. Перевірити доступ
2. **[Шлюз]** Джерело аудіо:
   - `audio_url` → проксі-байти з зовнішнього URL (обхід CORS)
   - Локальний файл (`audio_path`) → потокова передача з підтримкою Range
   - Нічого → Edge-TTS (генерація голосу) як fallback
3. Клієнт програє в Player

**Збереження прогресу:**
1. При закритті/паузі → `POST /api/books/{id}/progress`
2. Зберегти `text_position`, `audio_position`, `last_mode`, `completed`
3. **[Шлюз]** Книга прочитана (`completed=true`)?
   - Так → перевірити досягнення → Процес 7

---

## Процес 4: Публікація книги (Автор)

### Старт: Автор натискає "Додати книгу"

1. Автор заповнює форму (назва, опис, жанри, мова) та завантажує файли
2. `POST /api/books` (multipart: обкладинка + текстовий файл + аудіофайл)
3. Бекенд:
   - Зберегти обкладинку → `covers/`
   - Зберегти текст → `books/{uuid}.txt`
   - Зберегти аудіо → `audio/{uuid}.ext` (якщо є)
4. **[Шлюз]** Роль користувача = `user`?
   - Так → автоматично підвищити до `author`
   - Ні → залишити поточну роль
5. Створити `Book` зі статусом `draft`
6. Автор може публікувати → статус → `published`
7. **[Шлюз]** Книга позначена `is_premium=true`?
   - Так → доступна тільки передплатникам або підписникам автора
   - Ні → доступна всім

**Аналітика автора:**
- `GET /api/author/analytics/{book_id}` → перегляди, читання, прослуховування, завершення, вподобання, рейтинг
- `GET /api/author/summary` → зведена статистика по всіх книгах

---

## Процес 5: Оплата та підписка

### Підписка на платформу (Premium)

1. Користувач відкриває PaymentModal → `GET /api/subscriptions/plans`
2. Обрати план Premium
3. `POST /api/payments/platform/checkout` → створити `Payment` (статус `pending`), повернути `payment_id`
4. Користувач вводить дані картки
5. `POST /api/payments/platform/confirm` (card_number, exp_month, exp_year, cvc)
6. Бекенд:
   - Валідація алгоритмом Луна
   - **[Шлюз]** Картка тестова?
     - `4242...` → успіх
     - `...0002` → insufficient funds → 402
     - `...0069` → do not honor → 402
     - `...0119` → expired → 402
   - Успіх → `Payment.status = succeeded`
7. Активувати `UserSubscription` (план = `premium`, +30 днів)
8. Надіслати сповіщення користувачу

**Скасування підписки:**
1. `POST /api/payments/platform/cancel`
2. Понизити `UserSubscription` до `free`

### Підписка на автора

1. Користувач → `GET /api/payments/author-plan/{author_id}` → переглянути план
2. `POST /api/payments/author/{id}/checkout` → `Payment` (статус `pending`)
3. `POST /api/payments/author/{id}/confirm` → аналогічна валідація картки
4. Активувати `UserAuthorSub` (статус `active`, +30 днів)
5. Надіслати сповіщення

---

## Процес 6: Соціальна взаємодія

### Рецензування книги

1. Користувач читає книгу → натискає "Написати рецензію"
2. `POST /api/reviews` (book_id, rating 1-5, content)
3. **[Шлюз]** Рецензія вже існує від цього користувача?
   - Так → 409 Conflict
   - Ні → зберегти `Review`
4. Надіслати `Notification` власнику книги
5. Перевірити досягнення (кількість рецензій)

### Коментарі до рецензій

1. `POST /api/reviews/{id}/comments` (content, parent_id — для вкладених)
2. Зберегти `Comment`
3. Надіслати `Notification` автору рецензії

### Підписка на користувача

1. `POST /api/users/{id}/follow`
2. **[Шлюз]** Вже підписаний?
   - Так → відписатись (toggle)
   - Ні → створити `UserFollow`, надіслати сповіщення

### Скарги на контент

1. `POST /api/reports` (content_type, content_id, reason)
2. Зберегти `ContentReport` (статус `open`)
3. Адмін переглядає → `PUT /api/reports/{id}` → `resolved` або `dismissed`

---

## Процес 7: Система досягнень

### Тригери перевірки досягнень

Виклик `evaluate_achievements(db, user)` відбувається після:
- Завершення книги
- Написання рецензії
- Підписки на автора/платформу
- `GET /api/achievements` (manual refresh)

**Логіка:**
1. Завантажити всі `Achievement` з БД
2. Для кожного досягнення:
   - **[Шлюз]** Тип умови (`condition_type`):
     - `books_completed` → рахувати завершені книги
     - `reviews_written` → рахувати рецензії
     - `favorites` → рахувати вподобані
     - інші типи...
   - **[Шлюз]** Значення ≥ `condition_value` та ще не отримано?
     - Так → зберегти `UserAchievement`, надіслати сповіщення
3. Повернути список всіх досягнень із прапорцем `earned`

---

## Процес 8: Адміністративна модерація

### Старт: Адмін відкриває панель

**Управління користувачами:**
1. `GET /api/admin/users` → список
2. `PUT /api/admin/users/{id}` → змінити `is_active` або `role`
3. **[Шлюз]** Деактивація?
   - Так → заблокувати логін (усі наступні запити → 401)

**Модерація книг:**
1. `GET /api/admin/books` → всі книги
2. `DELETE /api/admin/content/book/{id}` → статус `banned`
3. `POST /api/admin/content/book/{id}/unban` → статус `published`

**Обробка скарг:**
1. `GET /api/admin/reports` → відкриті скарги
2. Переглянути контент
3. `PUT /api/admin/reports/{id}` → статус `resolved` або `dismissed`
4. При `resolved`: `DELETE /api/admin/content/{type}/{id}`

## Процес 9: Фонове наповнення контентом (Scraping)

### Startup Lifecycle (при старті бекенду)

```
Старт FastAPI
  ↓
Alembic міграції (sync)
  ↓
Seed досягнень (sync)
  ↓
Seed системного адміна (sync)
  ↓
[Паралельно у background tasks:]
  ├── catalog_warmup: live_sync_books("популярні", "фентезі", ...) + populate_audio_urls()
  ├── run_scraper_forever: Open Library + Gutendex кожні 30 хв
  └── run_booknet_forever: BookNet.ua scraper кожні BOOKNET_INTERVAL_MINUTES хв
```

### BookNet.ua Scraper

1. Логін на booknet.ua (httpx сесія)
2. Обхід каталогу `/top/all?page=N`
3. **[Шлюз]** Книга безкоштовна (`Безкоштовно`)?
   - Ні → пропустити
   - Так →
4. Зібрати метадані (назва, автор, жанри, обкладинка)
5. Читати повний текст сторінка за сторінкою (CSRF-токен + POST `/reader/get-page`)
6. Очистити невидимі DRM-символи
7. **[Шлюз]** Книга вже є в БД?
   - Так → пропустити
   - Ні → зберегти файл + запис `Book`
8. Затримка 1.5 с між запитами (rate limiting)

## Процес 10: Рекомендації

### Старт: `GET /api/recommendations`

1. Завантажити всі опубліковані книги з БД
2. **[Шлюз]** Є seed-книги користувача (вподобані + рецензії ≥4 + завершені)?
   - Ні → повернути trending (топ за переглядами)
   - Так →
3. Побудувати TF-IDF матрицю (назва + автор + опис + жанри + мова)
4. Усереднити вектори seed-книг → профіль користувача
5. Косинусна подібність профілю до всіх книг
6. **[Шлюз]** Спільні жанри є?
   - Так → додати бонус +0.3 до схожості
7. Виключити вже прочитані/вподобані книги
8. Повернути топ-12 з причиною (`content_based` / `collaborative`)

## Ключові сутності (Data Objects для BPMN)

```
User → [UserSubscription, UserAuthorSub, BookProgress, Review, 
         UserFollow, BookFavorite, UserAchievement, Notification, Payment]
Book → [BookProgress, Review, BookFavorite, BookEvent, ContentReport]
Review → [Comment, ContentReport]
Payment → [UserSubscription, UserAuthorSub]
Achievement → [UserAchievement]
AuthorSubPlan → [UserAuthorSub, Payment]
```

## Примітки для BPMN інструменту

- **Рекомендовані пули**: Гість, Користувач, Автор, Адмін, Зовнішні API (Stripe/BookNet/OpenLib)
- **Основні процеси для окремих діаграм**: Реєстрація, Доступ до книги, Оплата, Соціальна взаємодія, Модерація, Скрапінг
- **Тип нотації**: BPMN 2.0
- **Граничні події**: JWT expiry (timer), Payment timeout (timer), Scraper error (error boundary)
